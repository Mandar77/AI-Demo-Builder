üí° Key Architectural Insight

  • You (Person 2) = Stateful Coordinator - Stores all data, orchestrates workflow, tracks progress
  • Person 3 & 4 = Stateless Workers - Read from your DB, do work, update your DB

  • Phase 2-3 = Stateful (DynamoDB) - Your database coordinates async video processing

üè¢ Industry Best Practices

Approach 1: Microservices with Shared Database

What you have: Multiple services, one database

‚úÖ Pros

  • Simple data access - everyone reads/writes same DB
  • ACID transactions across services
  • Easy to query complete state
  • Fast for 18-day project
  • Common in startups & MVPs
  • Easy debugging - all data in one place

‚ùå Cons

  • Tight coupling via database schema
  • Schema changes affect everyone
  • Can't scale services independently
  • Database becomes bottleneck at scale
  • Requires coordination on schema

Approach 2: Event-Driven with Message Queue

Used by: Netflix, Uber, Amazon at scale

‚úÖ Pros

  • Loose coupling - services don't know about each other
  • Scale independently
  • Easy to add new services
  • Fault tolerant with retries
  • Industry standard at scale
  • Each service has own database

‚ùå Cons

  • Complex - need Kafka, RabbitMQ, etc.
  • Eventual consistency challenges
  • Harder to debug (distributed tracing needed)
  • Overkill for small projects
  • Steeper learning curve
  • More infrastructure to manage

üéØ Verdict for Your Project

Shared Database (Approach 1) is the RIGHT choice for you!

Your team size (5 people), timeline (18 days), and requirements (MVP demo) make shared database the optimal choice. Companies like Stripe, GitHub, and Airbnb started this way. You can always refactor to event-driven later if needed (YAGNI principle - You Ain't Gonna Need It).

Real Example: Instagram had 13 engineers and a shared PostgreSQL database when Facebook bought them for $1 billion. WhatsApp had 50 engineers and sold for $19 billion with a monolithic architecture. Don't over-engineer!
